Prismata AI Player Algorithms
=============================

This file explains the main full-player algorithms in source/ai and
source/ai/experimental.


Core Concepts
=============

GameState
---------

The complete current game state. A player receives a GameState and must produce
one complete Move for the active player.


Move
----

A sequence of Actions for the active player's turn. Most generated moves end by
advancing through the confirm phase.


PartialPlayer
-------------

A policy for one part of a turn: defense, action/ability, buy, or breach.
Several player algorithms use PartialPlayers as building blocks.


MoveIterator
------------

A generator of complete-turn candidate moves. Given a GameState, it repeatedly
returns:

    child state after candidate move
    candidate move

The main search players do not enumerate raw legal action sequences directly.
They search over complete-turn moves produced by MoveIterators.


Evaluation
----------

The search algorithms use one of these evaluation methods:

    Playout
        Finish the game using selected playout players and score the result.

    WillScore
        Evaluate the current state with a static card-value heuristic.

    WillScoreInflation
        Like WillScore, but gives the active player an initiative multiplier.


Player_PPSequence
=================

Idea
----

Compose one PartialPlayer per turn phase into a complete-turn policy. This is
not a tree search. It is a scripted sequence of phase policies.


Pseudocode
----------

    getMove(state):
        move = empty Move
        child = copy(state)

        for phasePolicy in phasePolicies:
            phaseMove = phasePolicy.getMove(child)
            child.doMove(phaseMove)
            move.addMove(phaseMove)

        append confirm/end action if needed
        return move


Strengths
---------

Fast and predictable. Good as a playout policy, baseline, or hand-authored
strategy.


Weaknesses
----------

It does not compare alternatives. Its quality is limited by the chosen phase
policies.


Player_Random
=============

Idea
----

Choose random legal actions until the turn is complete.


Pseudocode
----------

    getMove(state):
        move = empty Move
        child = copy(state)

        while child turn is not complete:
            legalActions = all legal actions in child
            action = random legal action
            child.doAction(action)
            move.addAction(action)

        return move


Strengths
---------

Useful as a sanity baseline.


Weaknesses
----------

The raw action space is noisy. Most random action sequences are strategically
poor.


Player_RandomFromIterator
=========================

Idea
----

Choose a random complete-turn move from a selected MoveIterator. This is a
cleaner random baseline than Player_Random because the iterator constrains the
choice to complete moves produced by phase policies.


Pseudocode
----------

    getMove(state):
        iterator = selected iterator clone
        iterator.getRandomMove(state, move)
        return move


Strengths
---------

Fast, simple, and samples from reasonable complete-turn moves.


Weaknesses
----------

It does not evaluate candidates. A strong move and a weak move from the same
iterator can be equally likely.


Player_AlphaBeta
================

Idea
----

Classic minimax alpha-beta search over complete-turn moves.


Pseudocode
----------

    getMove(state):
        bestMove = null

        for depth from 1 to MaxDepth while time remains:
            bestMove = alphaBetaRoot(state, depth)

        return bestMove

    alphaBeta(state, depth, alpha, beta):
        if state is terminal or depth == 0:
            return evaluate(state)

        if active player is max player:
            value = -infinity

            for child in MoveIterator(state):
                value = max(value, alphaBeta(child, depth - 1, alpha, beta))
                alpha = max(alpha, value)

                if alpha >= beta:
                    break

            return value

        else:
            value = +infinity

            for child in MoveIterator(state):
                value = min(value, alphaBeta(child, depth - 1, alpha, beta))
                beta = min(beta, value)

                if alpha >= beta:
                    break

            return value


Strengths
---------

Principled adversarial search. Alpha-beta pruning can avoid large parts of the
tree when move ordering is good.


Weaknesses
----------

The branching factor is still the core problem. If the root or opponent response
iterator has many complete-turn moves, depth becomes expensive quickly.


Player_StackAlphaBeta
=====================

Idea
----

An explicit-stack alpha-beta implementation. It searches the same kind of game
tree as Player_AlphaBeta, but avoids normal recursive call structure so it can
track time and partial progress more directly.


Pseudocode
----------

    getMove(state):
        search = StackAlphaBetaSearch(params)
        search.doSearch(state, move)
        return move

    doSearch(initialState):
        start timer

        for currentDepth from 1 to MaxDepth:
            initialize stack with root state
            result = stackAlphaBeta()

            if result completed:
                store best move at currentDepth

                if MaxDepth reached or no additional work was found:
                    break

            if result timed out:
                break

        return best move from deepest completed depth

    stackAlphaBeta():
        while stack is active:
            frame = top stack frame

            if time expired:
                return timeout

            if frame is terminal or at depth limit:
                value = evaluate(frame.state)
                pop frame and back up value
                continue

            child = frame.iterator.generateNextChild()

            if child exists:
                push child frame
                continue

            no children remain:
                pop frame and back up alpha/beta result

            if alpha >= beta:
                prune remaining siblings


Strengths
---------

This is the main strong adversarial-search structure. It handles time control
well and supports iterative deepening.


Weaknesses
----------

It still relies heavily on the iterator providing a compact and well-ordered
move set. Huge branching factor can prevent it from reaching meaningful depth.


Player_UCT
==========

Idea
----

Monte Carlo Tree Search using UCT selection. It builds a tree of complete-turn
moves, evaluates leaves by playout, and backs up win counts.


Pseudocode
----------

    getMove(state):
        root = UCTNode(state)

        while time and traversal budget remain:
            winner = traverse(root)
            back up winner through visited nodes

        return selected root child

    traverse(node):
        if node is non-root and unvisited:
            winner = playout(node.state)
            node.visits += 1
            node.wins += score(winner)
            return winner

        if node.state is terminal:
            winner = node.state.winner
            node.visits += 1
            node.wins += score(winner)
            return winner

        node.generateNextChild()
        child = selectChildByUCT(node)
        winner = traverse(child)

        node.visits += 1
        node.wins += score(winner)
        return winner

    selectChildByUCT(node):
        for child in node.children:
            if child.visits == 0:
                return child

            winRate = child.wins / child.visits
            exploration = C * sqrt(log(node.visits) / child.visits)
            value = winRate + exploration

        return child with highest value


Strengths
---------

Balances exploration and exploitation. Does not require full-width expansion of
the tree before it can start learning from playouts.


Weaknesses
----------

The existing implementation still uses a fixed MaxChildren cap. That cap is a
manual answer to branching factor rather than an adaptive one.


Player_RootRacing
=================

Idea
----

Spend the search budget at the root. Generate candidate complete-turn moves,
seed them with a cheap evaluation, then repeatedly evaluate and eliminate weaker
candidates.


Pseudocode
----------

    getMove(state):
        candidates = generateCandidates(state)

        if candidates is empty:
            return first generated legal move

        for candidate in candidates:
            candidate.seedScore = seedEvaluate(candidate.childState)

        sort candidates by seedScore

        for pass from 1 to MaxRacePasses while time remains:
            for candidate in candidates while time remains:
                candidate.scoreSum += evaluate(candidate.childState)
                candidate.visits += 1

            sort candidates by average score
            keep top KeepPerRound fraction

        return best candidate.move

    generateCandidates(state):
        for selected root iterator:
            generate child moves until MaxCandidates or timeout
            ignore duplicate moves and isomorphic child states


Strengths
---------

Avoids opponent-response explosion. It can compare many root moves without
committing to a deep game tree.


Weaknesses
----------

It is shallow. If a move only looks good after considering the opponent's best
reply, root racing may overvalue it.


Player_LazyUCT
==============

Idea
----

Tree search over complete-turn moves with lazy child generation. It is similar
to UCT, but backs up numeric scores from the max player's perspective:

    win  = +1
    loss = -1
    draw = 0

It can also use static evaluation.


Pseudocode
----------

    getMove(state):
        root = LazyUCTNode(state)

        while time and traversal budget remain:
            traverse(root, depth = 0)

        return most visited root child

    traverse(node, depth):
        if node is non-root and unvisited:
            score = evaluate(node.state)
            node.addVisit(score)
            return score

        if node is terminal or depth >= MaxDepth:
            score = evaluate(node.state)
            node.addVisit(score)
            return score

        if node can generate more children:
            node.generateNextChild()

        if node has no children:
            score = evaluate(node.state)
            node.addVisit(score)
            return score

        child = selectChildByUCT(node)
        score = traverse(child, depth + 1)

        node.addVisit(score)
        return score

    selectChildByUCT(node):
        for child in node.children:
            if child.visits == 0:
                return child

            mean = child.scoreSum / child.visits
            exploration = C * sqrt(log(node.visits) / child.visits)

            if node.activePlayer is max player:
                value = mean + exploration
            else:
                value = -mean + exploration

        return child with highest value


Strengths
---------

Starts searching before all children are generated. It gives a direct comparison
point against Player_UCT while using score backup instead of win-count backup.


Weaknesses
----------

It still allows expansion up to a fixed MaxChildren cap. The cap controls the
branching factor, but it does not adapt to node importance.


Player_ProgressiveWideningUCT
=============================

Idea
----

Use the same search structure as Player_LazyUCT, but make child generation
adaptive. A node can only open more children as its visit count grows.


Progressive Widening Rule
-------------------------

    allowedChildren = floor(WidenC * visits ^ WidenAlpha)
    allowedChildren = max(1, allowedChildren)
    allowedChildren = min(allowedChildren, MaxChildren)

Example with WidenC = 2.0 and WidenAlpha = 0.5:

    visits = 1    -> about 2 children
    visits = 25   -> about 10 children
    visits = 100  -> about 20 children
    visits = 400  -> about 40 children


Pseudocode
----------

    getMove(state):
        root = LazyUCTNode(state)

        while time and traversal budget remain:
            traverse(root, depth = 0)

        return most visited root child

    traverse(node, depth):
        if node is non-root and unvisited:
            score = evaluate(node.state)
            node.addVisit(score)
            return score

        if node is terminal or depth >= MaxDepth:
            score = evaluate(node.state)
            node.addVisit(score)
            return score

        allowed = floor(WidenC * node.visits ^ WidenAlpha)
        allowed = clamp(allowed, 1, MaxChildren)

        if node.children.size < allowed:
            node.generateNextChild()

        if node has no children:
            score = evaluate(node.state)
            node.addVisit(score)
            return score

        child = selectChildByUCT(node)
        score = traverse(child, depth + 1)

        node.addVisit(score)
        return score


Strengths
---------

Designed for huge branching factors. Early visits compare a small number of
candidates. Frequently visited nodes gradually open more alternatives. This
lets the search spend more effort where the tree appears important.


Weaknesses
----------

Its behavior depends on WidenC and WidenAlpha. If widening is too slow, it may
miss important moves. If widening is too fast, it behaves like fixed-cap UCT.


Player_PlanEvolution
====================

Idea
----

Search over compact turn plans instead of legal move trees. A plan stores
preferences for buying, activating abilities, blocking, targeting, breaching,
and ending the action phase. The compiler repeatedly asks the GameState for
legal actions and chooses the legal action that best matches the plan.


Pseudocode
----------

    getMove(state):
        population = seedPlans(state)

        while time remains:
            for plan in population that is not scored:
                move = compilePlan(state, plan)
                child = state after move
                plan.score = evaluate(child)

            sort population by score
            elites = best plans
            nextPopulation = elites

            while nextPopulation is not full:
                parentA = random elite
                parentB = random elite
                child = crossover(parentA, parentB)
                mutate(child)
                nextPopulation.add(child)

            population = nextPopulation

        return best plan's compiled move

    compilePlan(state, plan):
        move = empty Move
        child = copy(state)

        while child turn is not complete:
            actions = child.generateLegalActions()
            add END_PHASE if legal

            bestAction = action with highest plan score
            child.doAction(bestAction)
            move.addAction(bestAction)

        return move


Strengths
---------

It does not enumerate complete-turn moves first. It searches a smoother space
of strategic intent, then repairs each plan into a legal turn through the game
rules.


Weaknesses
----------

The compiler is greedy once a plan is chosen. A strong strategic plan can still
compile into a poor tactical sequence if the action scoring misses an important
interaction.


Algorithm Comparison
====================

PPSequence
    Scripted phase policy. Fast, no search.

RandomFromIterator
    Random complete-turn move from a structured iterator.

StackAlphaBeta
    Adversarial minimax search. Strong when branching is controlled and move
    ordering is good.

UCT
    Tree search using playout statistics and exploration bonuses.

RootRacing
    Flat root candidate racing. Avoids tree branching by spending effort only
    on root moves.

LazyUCT
    UCT-style tree search with lazy complete-turn child generation and numeric
    score backup.

ProgressiveWideningUCT
    LazyUCT with visit-based child expansion. Intended to manage very large
    move spaces in real time.

PlanEvolution
    Evolves strategic turn plans and compiles them directly through legal
    actions, avoiding complete-turn move enumeration.
